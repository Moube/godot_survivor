# Top-Down Shooter Demo Project Plan

## 目标

在当前 `Godot 4.6 + C#` 项目中，实现一个可玩的 2D Top-Down Shooter Demo，覆盖以下最小闭环：

1. 玩家移动
2. 朝鼠标方向瞄准与射击
3. 敌人生成与追踪
4. 子弹命中、伤害、死亡
5. 分数统计
6. 游戏结束与重新开始

本计划按“先可运行，再逐步完善”的顺序推进，避免前期过度设计。

---

## 实施原则

1. 优先建立可运行版本，再做结构优化。
2. 优先使用 Godot 的场景、节点、Inspector 和信号机制，不先做重框架。
3. C# 代码只负责逻辑，不强行把所有资源配置都代码化。
4. 每个阶段都要求有可验证结果。

---

## 目录建议

建议在项目中使用如下结构：

```text
res://
  scene/
    main/
    level/
    player/
    enemy/
    bullet/
    ui/
  asset/
  script/
    common/
```

推荐继续采用“场景与脚本就近放置”的方式，例如：

```text
res://scene/main/Main.tscn
res://scene/main/Main.cs
res://scene/level/Level01.tscn
res://scene/level/Level01.cs
res://scene/player/Player.tscn
res://scene/player/Player.cs
```

## 阶段 0：环境与项目初始化

### 目标

确认项目具备最基本的 Godot + C# 开发能力。

### 任务

1. 用 Godot 打开当前项目。
2. 确认 C# 支持正常，能够创建并挂载 C# 脚本。
3. 创建主场景 `Main.tscn`。
4. 将 `Main.tscn` 设置为主启动场景。
5. 确认可以运行空场景。

### 验收标准

1. Godot 可以正常打开项目。
2. 可以创建 C# 脚本且无编译错误。
3. 点击运行后能进入 `Main.tscn`。

### AI 可实现

1. 编写项目目录规划说明。
2. 生成基础 C# 脚本模板。
3. 说明 Godot 中需要设置哪些项目参数。

### 需要你手动操作

1. 确认本机 .NET / Godot C# 环境已正确安装。
2. 在 Godot 编辑器中打开项目。
3. 创建场景与节点。
4. 在 Project Settings 中设置 Main Scene。
5. 在编辑器顶部选项卡中找到 项目 > 工具 > C#，执行 `Create C# Solution`，生成 C# 工程文件。
6. 在编辑器右上角执行 `Build`，确认 C# 项目可以成功编译。

## 阶段 1：玩家基础移动

### 目标

建立最小可玩的基础流程：从开始菜单进入关卡，在关卡中生成玩家，并完成可视化移动、镜头跟随与基础碰撞验证。

### 任务

1. 创建 `Player.tscn`。
2. 使用 `CharacterBody2D` 作为玩家根节点。
3. 添加玩家可视节点与碰撞节点：
   - `CollisionShape2D`
   - 临时可视节点（如 `Polygon2D` 或简单占位图形）
   - `Camera2D`
4. 配置输入映射：
   - `move_up`
   - `move_down`
   - `move_left`
   - `move_right`
   - `fire`（为后续阶段预留）
5. 编写 `Player.cs` 中的基础移动逻辑。
6. 将 `Main.tscn` 调整为开始菜单场景。
7. 新建实际游玩的关卡场景，例如 `scene/level/Level01.tscn`。
8. 由关卡场景在出生点实例化并加载 `Player.tscn`。
9. 在关卡中加入临时背景参考元素，便于肉眼验证镜头跟随与位移。
10. 在关卡中加入静态障碍物与边界墙，用于碰撞验收。

### 验收标准

1. 运行项目后先进入开始菜单。
2. 点击开始按钮后能进入关卡场景。
3. 玩家可以四方向或八方向移动。
4. 镜头跟随正常，能通过背景参考明显看出位移。
5. 玩家不能穿过关卡边界和测试障碍物，贴边移动表现正常。

### AI 可实现

1. `Player.cs` 的移动逻辑实现。
2. 输入读取与速度控制代码。
3. `Main.tscn` 菜单跳转逻辑。
4. `Level01.tscn` / `Level01.cs` 的关卡组织与玩家实例化逻辑。
5. 临时可视化占位方案与碰撞测试场景配置建议。
6. 对 `_PhysicsProcess`、`Velocity`、`MoveAndSlide()` 用法进行解释。

### 需要你手动操作

1. 在 Godot 中创建 `Player.tscn` 节点树。
2. 在 Inspector 中配置 `CollisionShape2D` 形状。
3. 在 Project Settings > Input Map 中配置输入。
4. 运行项目，验证移动、镜头与碰撞表现是否符合预期。

## 阶段 2：瞄准与射击

### 目标

实现玩家朝鼠标方向瞄准并发射子弹。

### 任务

1. 为玩家增加朝向逻辑。
2. 创建 `Bullet.tscn`。
3. 使用 `Area2D` 或 `CharacterBody2D` 作为子弹节点。
4. 给玩家添加发射点 `Marker2D`。
5. 鼠标点击时实例化子弹。
6. 子弹朝目标方向飞行。
7. 子弹超出范围后销毁。
8. 增加基础射击间隔。

### 验收标准

1. 鼠标点击可以稳定发射子弹。
2. 子弹方向正确。
3. 子弹不会无限留在场景中。

### AI 可实现

1. `Bullet.cs` 移动与生命周期逻辑。
2. `Player.cs` 的射击逻辑。
3. 射击冷却时间控制代码。
4. 场景实例化与节点引用代码。

### 需要你手动操作

1. 创建 `Bullet.tscn`。
2. 配置子弹节点的碰撞区域和图形。
3. 在玩家场景中添加 `Marker2D` 作为枪口/发射点。
4. 在 Input Map 中添加射击输入，例如 `shoot`。

---

## 阶段 3：敌人与追踪

### 目标

让敌人进入场景并持续追击玩家。

### 任务

1. 创建 `Enemy.tscn`。
2. 使用 `CharacterBody2D` 作为敌人根节点。
3. 添加基础外观与碰撞。
4. 实现敌人朝玩家移动。
5. 在关卡场景中放置一个或多个敌人。
6. 后续增加简单刷怪器 `Spawner`。

### 验收标准

1. 敌人进入场景后会持续朝玩家移动。
2. 敌人与玩家距离变化符合预期。
3. 同时存在多个敌人时逻辑稳定。

### AI 可实现

1. `Enemy.cs` 的追踪逻辑。
2. 敌人移动参数设计建议。
3. 简单刷怪器逻辑实现。
4. 玩家引用获取方式设计。

### 需要你手动操作

1. 创建 `Enemy.tscn` 节点树。
2. 配置敌人的碰撞形状和图形资源。
3. 在关卡场景中摆放初始出生点，或创建生成器节点。

## 阶段 4：伤害、死亡与分数

### 目标

建立核心战斗闭环：命中、扣血、销毁、加分。

### 任务

1. 给敌人添加生命值。
2. 子弹命中敌人后造成伤害。
3. 敌人死亡后销毁。
4. 玩家被敌人接触或攻击后掉血。
5. 玩家血量归零后进入 Game Over。
6. 引入分数系统。
7. 击杀敌人时增加分数。

### 验收标准

1. 子弹命中敌人后血量正确减少。
2. 敌人死亡时从场景移除。
3. UI 或日志中可看到分数变化。
4. 玩家死亡后不再可继续正常游戏。

### AI 可实现

1. 生命值、伤害与死亡的代码实现。
2. 子弹与敌人碰撞处理逻辑。
3. 分数管理逻辑。
4. `GameManager` 或 `Autoload` 的简化实现建议。

### 需要你手动操作

1. 在 Godot 中配置碰撞层和掩码。
2. 连接或确认碰撞/信号节点设置正确。
3. 手动运行游戏验证碰撞行为。

## 阶段 5：UI 与游戏流程

### 目标

补齐演示所需的基础界面和流程控制。

### 任务

1. 创建 HUD：
   - 玩家血量
   - 当前分数
   - 当前击杀数（可选）
2. 创建 Game Over UI。
3. 添加重新开始按钮。
4. 完善开始界面与流程跳转。
5. 完成“开始 -> 游戏中 -> 结束 -> 重开”的流程闭环。

### 验收标准

1. HUD 信息能随游戏状态变化更新。
2. 死亡后显示 Game Over。
3. 点击重开后可以重新开始一局。

### AI 可实现

1. UI 更新逻辑代码。
2. 游戏状态枚举与流程控制代码。
3. 重开逻辑实现。
4. HUD 脚本与主控制逻辑。

### 需要你手动操作

1. 在 Godot 中创建 UI 场景与控件树。
2. 调整锚点、布局与显示层级。
3. 在 Inspector 中绑定必要的节点引用。

## 阶段 6：打磨与最小优化

### 目标

将 Demo 从“能跑”提升到“可演示”。

### 任务

1. 加入基础音效：
   - 射击
   - 命中
   - 死亡
2. 增加基础视觉反馈：
   - 子弹命中闪烁
   - 死亡消失效果
   - 屏幕震动（可选）
3. 调整数值：
   - 玩家速度
   - 子弹速度
   - 射击频率
   - 敌人速度
   - 刷怪频率
4. 修复明显体验问题：
   - 出生点过近
   - 子弹穿透异常
   - UI 显示错位

### 验收标准

1. 游戏节奏基本合理。
2. 音画反馈能支撑演示。
3. 无明显阻断体验的功能问题。

### AI 可实现

1. 参数调优建议。
2. 小型反馈逻辑代码。
3. 简单特效触发代码。
4. 问题排查与重构建议。

### 需要你手动操作

1. 导入音效与贴图资源。
2. 在编辑器中调试节点布局和资源绑定。
3. 实机试玩并反馈手感问题。

## 推荐开发顺序

建议严格按以下顺序推进，不要跳步：

1. `Main.tscn` 跑起来
2. `Level01.tscn` 可进入
3. `Player.tscn` 可移动
4. `Bullet.tscn` 可发射
5. `Enemy.tscn` 可追踪
6. 子弹命中敌人
7. 玩家受伤与死亡
8. 分数系统
9. HUD 与 Game Over
10. 刷怪与数值调整
11. 音效和演示打磨

原因：

1. 每一步都能独立验证。
2. 出问题时定位范围小。
3. 符合 Godot 项目从场景到逻辑的自然开发方式。

## AI 与人工分工原则

### 适合交给 AI 的部分

1. C# 脚本编写
2. 场景结构建议
3. 节点通信方案
4. 信号连接方式建议
5. 功能重构建议
6. 调试思路梳理
7. 小型工具类和通用逻辑实现
8. 文档、注释、阶段复盘

### 更适合你手动完成的部分

1. 在 Godot 编辑器中创建场景和节点树
2. 资源导入与替换
3. 碰撞体、锚点、层级、Mask、Inspector 参数配置
4. 游戏手感验证
5. 数值调整
6. 美术和 UI 的最终布局
7. 对设计意图和玩法取舍做决定

### 不建议完全交给 AI 的部分

1. 游戏手感判断
2. 镜头、UI、移动速度等演示体验的最终定稿
3. 需求边界与玩法范围控制

原因：

1. AI 可以生成实现，但不能替代你对“是否好玩、是否顺手”的判断。
2. Godot 项目中大量问题发生在编辑器配置和资源绑定层，这部分需要你实际操作验证。

---

## 每阶段建议工作流

每个功能点都建议按这个顺序执行：

1. 你先在 Godot 中创建或调整节点结构。
2. AI 根据该节点结构编写对应 C# 脚本。
3. 你在编辑器中挂载脚本、配置 Inspector 参数。
4. 运行并验证结果。
5. AI 根据报错、现象和日志继续修复。
6. 功能稳定后再进入下一阶段。

## 结论

当前项目最合适的做法，不是先搭复杂架构，而是围绕 Top-Down Shooter 的最小闭环逐步推进。  
AI 负责高频代码生成、结构建议、排错和重构；你负责 Godot 编辑器中的场景搭建、资源配置、碰撞参数、UI 布局和玩法判断。

只要严格按阶段推进，这个项目会很适合作为你熟悉 Godot 和 C# 的第一个实践样本。
