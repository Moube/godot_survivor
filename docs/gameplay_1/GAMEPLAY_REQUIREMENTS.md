# Gameplay 1 需求文档

## 文档目的

本文件用于重新定义当前项目下一轮玩法方向，作为后续系统实现、UI 交互设计和 UI 资源生成的依据。

当前项目已经具备 Top-Down Shooter 的最小闭环，但原有 UI 只服务于分数、血量和 Game Over。新的玩法方向已经从“射击演示 Demo”转向“轻量幸存者玩法”，因此需要先重新明确局内系统，再继续推进 UI 与美术资源。

---

## 核心定位

本轮玩法目标是实现一个 **轻量幸存者式 Top-Down Shooter**：

- 玩家控制小法师在关卡中移动。
- 武器自动攻击，不需要玩家手动点击射击。
- 不同武器拥有不同瞄准逻辑，例如跟随鼠标、自动索敌或随机方向。
- 敌人持续生成，随存活时间推进逐渐提高压力。
- 敌人死亡后掉落经验宝石或经验粒子。
- 玩家需要主动靠近经验掉落物才能拾取。
- 经验满后升级，并从 3 个随机奖励中选择 1 个。
- 奖励可能是新武器、已有武器升级或被动物品。
- 局内目标是尽可能长时间存活。
- 局内不显示分数。
- 结算暂时只显示存活时间。

---

## 局内核心循环

第一版局内循环如下：

1. 玩家进入关卡。
2. 存活计时开始。
3. 敌人按照时间配置持续生成。
4. 玩家通过移动躲避敌人。
5. 已持有武器按各自规则自动攻击。
6. 敌人死亡后掉落经验宝石。
7. 玩家靠近经验宝石并拾取。
8. 经验达到升级阈值后暂停或冻结局内战斗。
9. 显示 3 个升级选项。
10. 玩家选择 1 个奖励。
11. 奖励立即生效，游戏继续。
12. 玩家死亡后进入结算界面。
13. 结算显示存活时间，并确认退出到主页面。

---

## 玩家控制

### 移动

- 使用键盘控制八方向移动。
- 当前可沿用已有移动输入。
- 第一版不强制加入冲刺、翻滚或体力系统。

### 瞄准

- 鼠标位置仍然有意义，但不再代表手动射击。
- 对于 `MouseDirection` 类型武器，鼠标决定攻击方向。
- 对于自动索敌、随机方向等武器，鼠标不参与其攻击方向。

### 射击

- 第一版取消手动射击输入作为核心玩法。
- 武器由自身冷却、目标规则和等级参数决定何时攻击。

---

## 武器系统

### 基础规则

- 本项目中的“武器”应理解为局内自动生效的战斗能力，不一定是具象的枪、法杖或发射器。
- 武器可以表现为投射物、范围区、光环、环绕物、周期脉冲、从天而降的区域效果等。
- 玩家可以持有多个武器。
- 每个武器独立计时、独立触发效果。
- 武器可以通过升级获得，也可以对已有武器进行等级提升。
- 第一版建议最大持有武器数量为 4 个。
- 武器实现应拆分为“武器运行时状态”和“具体效果行为”，避免把所有武器都写成发射子弹。

### 武器行为类型

第一版应先支持一种最基础的武器行为：`ProjectileEmitter`，即存在一个发射器并发射投射物的武器。

后续武器行为可以扩展为：

| 行为类型 | 说明 |
| --- | --- |
| `ProjectileEmitter` | 存在发射点，周期性发射投射物 |
| `GroundArea` | 在地面生成持续一段时间的范围效果，例如圣水池 |
| `PlayerAura` | 以玩家为中心的持续光环或伤害圈 |
| `OrbitingObject` | 围绕玩家旋转的实体或伤害源 |
| `AreaPulse` | 以玩家或目标点为中心周期性爆发 |
| `TargetedStrike` | 在敌人或随机区域上方/脚下生成打击效果 |

### 投射物发射行为

之前规划的 3 种模式不是所有武器的通用类型，而是 `ProjectileEmitter` 类武器在发射投射物时的方向/目标规则。

`ProjectileEmitter` 第一版至少支持以下发射行为：

| 模式 | 说明 |
| --- | --- |
| `MouseDirection` | 按鼠标方向发射，适合当前浮游魔法短杖 |
| `NearestEnemy` | 自动选择最近敌人发射 |
| `RandomDirection` | 每次向随机方向发射 |

后续可扩展：

| 模式 | 说明 |
| --- | --- |
| `ForwardSweep` | 朝某方向扇形扫射 |
| `MultipleNearest` | 同时向多个近距离敌人发射 |
| `FixedPattern` | 按固定角度或序列发射 |

### 第一版武器内容建议

第一版不需要一次性做很多武器。建议先实现 2 到 3 个，用来验证系统。

第一批可以全部基于 `ProjectileEmitter` 实现，但接口和命名需要允许后续加入非发射器武器。

1. 浮游魔法短杖
   - 行为类型：`ProjectileEmitter`
   - 发射行为：`MouseDirection`
   - 表现：朝鼠标方向发射魔法弹
   - 来源：初始武器

2. 星火法球
   - 行为类型：`ProjectileEmitter`
   - 发射行为：`NearestEnemy`
   - 表现：周期性向最近敌人发射小型法球
   - 来源：升级奖励

3. 混沌飞弹
   - 行为类型：`ProjectileEmitter`
   - 发射行为：`RandomDirection`
   - 表现：随机方向发射飞弹
   - 来源：升级奖励

后续建议补充非发射器武器，例如：

- 圣水池：在随机敌人附近或玩家附近生成持续伤害区域。
- 魔力光环：以玩家为中心持续影响附近敌人。
- 落雷：周期性选择敌人位置进行延迟打击。

## 被动物品系统

### 基础规则

- 被动物品用于提升玩家或全局战斗属性。
- 被动物品通过升级奖励获得。
- 第一版建议最大持有被动数量为 4 个。
- 被动可以有等级。

### 第一版被动内容建议

| 被动物品 | 效果 |
| --- | --- |
| 轻盈靴 | 提升移动速度 |
| 魔力透镜 | 提升全部武器伤害 |
| 快速咒文 | 降低全部武器冷却 |
| 收集护符 | 提升经验拾取范围 |
| 生命护符 | 提升最大生命值 |

---

## 敌人与刷怪

### 生成规则

- 不显示“波次”概念。
- 局内持续刷怪。
- 内部使用时间节点配置控制刷怪强度。
- 时间推进后，可改变：
  - 可生成的敌人类型
  - 单次生成数量
  - 生成间隔
  - 场上敌人数量上限
  - 敌人权重

### 第一版敌人类型

| 敌人 | 定位 |
| --- | --- |
| 普通史莱姆 | 基础追踪敌人 |
| 快速史莱姆 | 低血量、高速度，用于制造移动压力 |
| 大史莱姆 | 高血量、低速度，用于制造空间压力 |

第一版可以先使用同一套基础敌人场景和不同参数配置验证玩法，不要求立即生成完整新美术。

---

## 经验掉落与拾取

### 掉落

- 敌人死亡后掉落经验宝石或经验粒子。
- 第一版可以只掉落一种经验物。
- 后续可扩展不同经验价值的掉落物。

### 拾取

- 玩家不会自动获得全地图经验。
- 玩家需要移动到经验物附近，进入拾取范围后获得经验。
- 拾取范围应作为玩家属性，允许被动物品提升。

### 经验表现

- 第一版经验物应有明确可读性。
- 不要求复杂吸附动画，但建议后续加入靠近后飞向玩家的表现。

---

## 升级三选一

### 前置依赖

升级三选一不是纯 UI 功能。实现前应先具备以下运行时基础：

- `WeaponInventory` 能记录玩家当前持有武器、等级、最大持有数量，并能添加新武器或升级已有武器。
- `PassiveInventory` 能记录玩家当前持有被动、等级、最大持有数量，并能添加新被动或升级已有被动。
- `PlayerStats` 或等价属性层能集中计算玩家移动速度、最大生命、拾取范围、武器伤害倍率、武器冷却倍率等可被被动影响的属性。
- 至少存在可实际生效的武器奖励和被动奖励，避免升级界面只改变文字状态。
- 经验曲线、升级池、武器定义和被动定义应来自配置数据。

### 触发

- 玩家经验达到当前等级阈值后触发升级。
- 触发升级时暂停或冻结局内战斗，避免玩家在选择期间受伤。

### 选择

- 每次升级显示 3 个候选奖励。
- 奖励从升级池中随机抽取。
- 候选奖励可以是：
  - 新武器
  - 已持有武器升级
  - 新被动
  - 已持有被动升级

### 奖励池约束

- 如果武器栏已满，则不再提供新武器，只提供已有武器升级或被动奖励。
- 如果被动栏已满，则不再提供新被动，只提供已有被动升级或武器奖励。
- 如果某个武器或被动已达到最大等级，则不再提供其升级奖励。

## 局内 UI 需求

第一版 HUD 应服务玩法信息，不追求装饰复杂度。

### 必须显示

1. 存活时间
   - 建议显示在顶部中央。
   - 格式：`mm:ss`。

2. 玩家生命
   - 建议保留底部中央横向血条。
   - 第一版不显示具体数字。

3. 经验条
   - 建议放在生命条附近。
   - 需要清楚表达当前等级进度。

4. 已持有武器列表
   - 建议放在屏幕左侧。
   - 第一版可以使用图标槽或文字占位。

5. 已持有被动列表
   - 建议放在屏幕右侧。
   - 第一版可以使用图标槽或文字占位。

6. 升级选择界面
   - 居中显示 3 个选项。
   - 每个选项至少包含名称、类型和简短效果说明。

7. 结算界面
   - 显示存活时间。
   - 提供确认按钮，返回主页面。

### 不显示

- 不显示分数。
- 第一版不显示波次文本。
- 第一版不显示伤害数字。
- 第一版不做复杂技能栏冷却表现。

## 主页面与局外流程

### 主菜单

主菜单需要包含：

1. 开始游戏
2. 设置
3. 退出游戏

### 关卡选择

- 点击开始游戏后进入关卡选择。
- 当前只有一个关卡。
- 关卡选择仍应独立存在，为后续新增关卡留结构。

### 设置

第一版设置可以先做占位，或只提供最基础选项。

建议后续支持：

- 主音量
- 音效音量
- 全屏开关
- 返回按钮

### 退出游戏

- 桌面导出版本中退出游戏。
- 编辑器运行时可调用 Godot 的退出逻辑，或保留按钮行为待验证。

---

## 结算需求

玩家死亡后进入结算。

第一版只显示：

- `Survived mm:ss`
- 确认按钮

确认后返回主页面。

后续可扩展：

- 击杀数量
- 达到等级
- 获得武器和被动
- 击败敌人统计
- 本局评价

---

## 内容数据需求

本轮系统需要把以下内容作为可调整数据纳入后续实现范围：

1. 武器定义
2. 被动定义
3. 敌人定义
4. 刷怪时间表
5. 升级奖励池
6. 玩家初始属性
7. 经验曲线
8. 关卡入口配置

这些数据用于调参和内容扩展，不记录本局运行状态。

### 配置访问入口

本轮需要一个统一的配置访问入口，用于按 ID 获取武器、被动、敌人、关卡、刷怪表和升级池等内容数据。

该入口主要服务以下场景：

1. 主菜单和关卡选择界面获取可进入关卡列表。
2. 进入关卡时，根据关卡 ID 获取对应关卡数据。
3. 刷怪系统根据敌人 ID 获取敌人数据。
4. 武器系统根据武器 ID 获取武器数据。
5. 升级系统根据奖励池中的 ID 解析具体武器或被动。
6. 后续存档或局外进度可以只保存内容 ID，再通过配置访问入口还原具体数据。

该入口不负责记录本局状态，也不直接控制刷怪、升级、武器攻击或玩家属性变化。

---

## 非目标

以下内容不进入第一版 Gameplay 1：

1. 多角色选择
2. 局外永久成长
3. 商店系统
4. 背包或道具栏管理
5. Boss 战
6. 多关卡正式内容
7. 复杂设置页面
8. 完整图标美术资源
9. 复杂伤害公式
10. 伤害数字
11. 屏幕震动
12. 手动射击作为核心操作

---

## 第一版验收标准

Gameplay 1 完成时，应满足：

1. 进入关卡后显示存活时间、生命条和经验条。
2. 玩家不按射击键，初始武器也会自动攻击。
3. 至少存在 2 种不同战斗行为或发射行为的武器，其中第一版可先用不同 `ProjectileEmitter` 发射行为验证。
4. 敌人会随时间持续生成，并且后续时间节点压力明显提高。
5. 敌人死亡会掉落经验物。
6. 玩家靠近经验物后可以拾取并获得经验。
7. 经验满后显示 3 选 1 升级界面。
8. 选择升级后，奖励会立即影响玩家、武器或被动。
9. HUD 能显示已持有武器和被动。
10. 玩家死亡后显示存活时间结算。
11. 结算确认后返回主页面。
12. 主页面包含开始游戏、设置、退出游戏。
13. 开始游戏后先进入关卡选择，再进入关卡。
