# Gameplay 1 实现计划

## 文档目的

本文件用于规划 Gameplay 1 的实现顺序。目标是在现有 Top-Down Shooter Demo 的基础上，逐步演进为轻量幸存者玩法，而不是推倒重做。

对应需求见：

- [GAMEPLAY_REQUIREMENTS.md](/D:/project/godot/test_godot/docs/gameplay_1/GAMEPLAY_REQUIREMENTS.md)

## 当前基础

项目当前已经具备以下基础：

- `scene/player/Player.tscn` 与 `Player.cs`
- `scene/weapon/Weapon2D.cs`
- `scene/weapon/MagicWandWeapon.cs`
- `scene/bullet/Bullet.tscn` 与 `Bullet.cs`
- `scene/enemy/Enemy.tscn`、`Enemy.cs` 与 `EnemyBase.cs`

- `scene/ui/Hud.tscn` 与 `Hud.cs`
- `script/common/GameSession.cs`
- `script/common/CombatComponent.cs`

本轮应优先复用这些基础：

- 保留玩家移动、生命、死亡。
- 保留浮游魔法短杖作为初始武器原型。
- 保留敌人追踪和接触伤害。
- 保留 `GameSession` 作为局内运行状态入口，但需要扩展职责。
- 改造 HUD，而不是继续围绕分数 UI 打磨。

建议新增目录：

```text
res://script/config/
res://script/gameplay/
res://scene/pickup/
res://scene/ui/upgrade/

```

说明：

- `script/config` 放内容数据类型与配置访问入口。
- `script/gameplay` 放局内运行控制器。
- `asset/config` 放内容数据资源。
- 场景仍按现有方式靠近对应模块放置。

## UI 实现原则

本轮所有 UI 页面先在没有正式贴图资源的情况下实现，优先保证流程、布局、状态切换和信息可读性。

实现时需要为后续补贴图预留结构：

1. 面板、按钮、卡牌、图标槽等 UI 元素应使用稳定的节点层级和命名。
2. 不把文字、背景、边框和状态逻辑混在同一个临时节点中。
3. 升级卡、武器槽、被动槽、结算面板等重复 UI 应尽量做成可复用子场景。
4. 先使用 Godot 样式、纯色、边框和占位图标表达信息层级。
5. 后续补贴图时，应能主要通过替换 StyleBox、TextureRect、图标资源或主题资源完成，不重写 UI 流程逻辑。

## 阶段 1：局内目标改为存活时间

### 目标

移除“分数是局内核心目标”的设计，改为存活时间驱动。

### 任务

1. 扩展 `GameSession`：
   - `ElapsedRunTime`
   - `RunTimeChanged` 信号
   - `FinalSurvivalTime`
   - Game Over 信号改为传递存活时间

2. 调整 `Level01`：
   - 开局调用 `GameSession.StartNewRun()`
   - 每帧或固定间隔推进存活时间
   - 玩家死亡时调用 `TriggerGameOver()`，结算存活时间

3. 调整 `Hud`：
   - 暂时隐藏或移除分数显示
   - 显示顶部中央存活时间
   - Game Over 面板改为显示 `Survived mm:ss`
   - 先使用无贴图样式实现，但保留后续替换血条、面板和按钮贴图的节点结构

### 验收

1. 进入关卡后计时从 00:00 开始。
2. 玩家死亡后显示最终存活时间。
3. UI 不再显示分数。

## 阶段 2：内容数据与配置访问入口

### 目标

先明确武器、敌人、刷怪、升级和关卡入口需要哪些可调整数据，并建立统一配置访问入口，避免后续系统实现时继续把数值散落在脚本里。

### 任务

1. 梳理第一版需要的数据：
   - 玩家初始生命、移动速度、拾取范围
   - 初始武器
   - 武器名称、瞄准模式、冷却、伤害、弹体数量
   - 被动名称、影响属性、每级数值、最大等级
   - 敌人生命、速度、接触伤害、经验掉落值
   - 刷怪时间节点、生成间隔、生成数量、场上上限
   - 升级奖励池
   - 经验曲线
   - 关卡入口信息

2. 新增必要枚举：
   - `WeaponAimMode`
   - `UpgradeRewardType`
   - `PlayerStatType`

3. 新增 `GameConfigManager`：
   - 作为 Autoload 使用。
   - 负责加载和缓存内容数据。
   - 提供按 ID 查询接口。
   - 启动时检查重复 ID 和缺失引用。

4. `GameConfigManager` 第一版建议接口：
   - `GetWeaponConfig(string id)`
   - `GetPassiveConfig(string id)`
   - `GetEnemyConfig(string id)`
   - `GetLevelConfig(string id)`
   - `GetSpawnScheduleConfig(string id)`
   - `GetUpgradePoolConfig(string id)`
   - `GetAllLevelConfigs()`

5. 明确运行时读取边界：
   - 关卡入口负责提供本局需要的数据。
   - 武器系统只读取武器相关数据。
   - 刷怪系统只读取敌人和刷怪相关数据。
   - 升级系统只读取奖励池、武器、被动和经验曲线相关数据。
   - HUD 只读取运行时状态，不直接读取内容数据。
   - `GameConfigManager` 不记录本局状态，不直接生成敌人、武器或升级选项。

6. 第一轮允许部分内容先用临时导出参数或场景引用推进实现，但新增系统应优先通过 `GameConfigManager` 获取跨系统共享数据。

### 验收

1. 第一版需要的数据项已经列清楚。
2. `GameConfigManager` 可以通过 ID 获取武器、被动、敌人和关卡数据。
3. 主菜单或关卡选择可以通过 `GameConfigManager` 获取关卡列表。
4. 每个运行时系统应读取哪些数据已经划分清楚。
5. 后续实现不需要把武器、敌人、刷怪和升级数值继续硬编码到各处。

## 阶段 3：武器改为自动攻击

### 目标

把当前依赖射击输入的武器改造为自动攻击系统。

### 任务

1. 修改 `Weapon2D`：
   - 不再依赖外部持续调用 `SetFireRequested(true)`。
   - 根据武器启用状态、冷却和瞄准规则自动发起攻击。
   - 保留发射动画和子弹生成逻辑。

2. 引入瞄准模式：
   - `MouseDirection`
   - `NearestEnemy`
   - `RandomDirection`

3. 新增或调整武器控制层：
   - 玩家持有多个武器。
   - 每个武器由武器数据初始化。
   - 初始武器由关卡入口提供。

4. 改造 `MagicWandWeapon`：
   - 作为 `MouseDirection` 初始武器。
   - 保留围绕玩家旋转。
   - 朝鼠标方向自动发射。

### 验收

1. 不按鼠标，浮游短杖也会按冷却自动发射。
2. 鼠标移动会影响浮游短杖攻击方向。
3. 至少有一个测试武器可以自动索最近敌人。
4. 至少有一个测试武器可以随机方向发射。

## 阶段 4：时间驱动刷怪

### 目标

用时间表替代固定或简单刷怪逻辑，不显示波次，但内部按时间增强压力。

### 任务

1. 新增 `SpawnDirector`：
   - 读取刷怪时间表数据
   - 根据 `GameSession.ElapsedRunTime` 选择当前阶段
   - 控制生成间隔、生成数量和场上数量上限

2. 改造敌人生成：
   - 根据敌人数据实例化敌人场景
   - 生成后应用生命、速度、接触伤害、经验掉落值

3. 配置第一版时间表：
   - 0:00 普通史莱姆，低强度
   - 1:00 提高普通史莱姆生成量和上限
   - 2:00 加入快速史莱姆
   - 3:00 加入大史莱姆

### 验收

1. 不需要玩家操作，敌人会持续生成。
2. 存活时间越长，敌人压力越高。
3. UI 不显示波次。
4. 当前场上敌人数量不会无限增长到失控。

## 阶段 5：经验掉落与拾取

### 目标

建立击杀敌人后获取经验的基础循环。

### 任务

1. 新增经验物场景：
   - `scene/pickup/ExperienceGem.tscn`
   - `ExperienceGem.cs`

2. 敌人死亡时：
   - 不再加分。
   - 根据敌人数据中的经验值掉落经验物。

3. 玩家拾取：
   - 新增拾取范围属性。
   - 玩家靠近经验物后拾取。
   - 拾取后通知 `ExperienceController`。

4. 扩展 HUD：
   - 显示经验条。
   - 显示当前等级可选，第一版可以不显示等级数字。

### 验收

1. 敌人死亡后生成经验物。
2. 玩家必须靠近经验物才能拾取。
3. 拾取后经验条增加。
4. 经验物被拾取后从场景移除。

## 阶段 6：升级三选一

### 目标

经验满后暂停局内战斗，显示 3 个升级奖励，并应用选择结果。

### 任务

1. 新增 `ExperienceController`：
   - 当前等级
   - 当前经验
   - 当前升级所需经验
   - 经验满时发出升级请求

2. 新增 `UpgradeManager`：
   - 读取升级奖励池数据
   - 根据玩家当前武器和被动状态过滤奖励
   - 随机生成 3 个候选项
   - 应用玩家选择

3. 新增 UI：
   - `scene/ui/upgrade/UpgradeChoicePanel.tscn`
   - 三张选项卡
   - 每张卡显示名称、类型和简短效果
   - 先使用无贴图卡牌样式实现，但卡牌节点结构应支持后续替换背景、边框和图标贴图

4. 奖励应用：
   - 新武器：加入 `WeaponInventory`
   - 武器升级：提升对应武器等级
   - 新被动：加入 `PassiveInventory`
   - 被动升级：提升对应被动等级

### 验收

1. 经验满后出现 3 个升级选项。
2. 选择期间玩家不会继续受伤。
3. 点击奖励后界面关闭，战斗继续。
4. 奖励能实际改变玩家属性或武器表现。

## 阶段 7：武器与被动持有列表

### 目标

让玩家已持有的武器和被动成为明确的运行时状态，并接入 HUD。

### 任务

1. 新增或整理：
   - `WeaponInventory`
   - `PassiveInventory`
   - `PlayerStats`

2. 支持最大持有数量：
   - 武器最大 4 个
   - 被动最大 4 个

3. HUD 显示：
   - 左侧武器列表
   - 右侧被动列表
   - 第一版可用文字或临时图标占位
   - 武器槽和被动槽需要预留后续图标贴图位置

### 验收

1. 获得新武器后 HUD 列表更新。
2. 获得新被动后 HUD 列表更新。
3. 升级已有武器或被动后，等级显示更新。
4. 达到持有上限后，升级池不再提供对应类型的新物品。

## 阶段 8：主菜单、关卡选择与结算

### 目标

建立新的局外流程：主菜单 -> 关卡选择 -> 局内 -> 结算 -> 主菜单。

### 任务

1. 改造主菜单：
   - Start Game
   - Settings
   - Quit
   - 先使用无贴图按钮样式实现，但保留后续按钮贴图和主题替换空间

2. 新增关卡选择界面：
   - 当前只有 `Level01`
   - 选择后加载对应关卡，并传入该关卡所需的局内数据
   - 先使用无贴图列表或按钮实现关卡入口

3. 新增设置界面占位：
   - 第一版可只提供返回按钮
   - 后续再接入音量和全屏
   - 先完成页面结构，不制作正式设置 UI 资源

4. 改造结算界面：
   - 显示存活时间
   - Confirm 后返回主页面
   - 不再直接 Restart
   - 先使用无贴图面板实现，但保留后续结算面板贴图替换空间

### 验收

1. 主菜单按钮流程完整。
2. 点击开始游戏后进入关卡选择，而不是直接进关。
3. 关卡选择可以启动当前唯一关卡。
4. 玩家死亡后结算确认返回主页面。

## 阶段 9：数值调优与最小打磨

### 目标

让 Gameplay 1 具备可试玩的节奏，而不急于进入 UI 美术资源生成。

### 任务

1. 调整初始参数：
   - 玩家移动速度
   - 玩家生命值
   - 初始武器冷却
   - 初始武器伤害
   - 敌人速度和生命
   - 刷怪间隔和上限
   - 经验曲线

2. 检查体验问题：
   - 初期是否过难
   - 1 分钟后是否压力明显提高
   - 经验拾取是否需要移动决策
   - 升级奖励是否能被感知
   - 局内 UI 是否遮挡战斗

3. 保持 UI 灰模：
   - 只要信息结构正确，不急着做最终资源。

### 验收

1. 玩家可以自然完成至少 2 到 3 次升级。
2. 游戏压力随时间增强。
3. 自动武器、经验拾取、升级选择形成闭环。
4. HUD 信息足够支撑局内决策。

## 关键迁移点

### 分数系统

当前 `GameSession` 有 `Score` 和 `ScoreChanged`，敌人死亡会加分。Gameplay 1 中分数不再作为局内目标。

建议处理：

- 第一阶段先隐藏分数 UI。
- 后续将击杀奖励改为经验掉落。
- `Score` 可暂时保留但不显示，最终再决定删除或改为内部统计。

### Game Over 信号

当前 Game Over 传递最终分数。Gameplay 1 应改为传递存活时间。

建议：

- 新增存活时间字段。
- Game Over UI 改读存活时间。
- 避免继续把 `finalScore` 作为结算参数。

### 武器触发方式

当前 `Weapon2D` 支持外部请求开火。Gameplay 1 应改为武器自动触发。

建议：

- 保留发射动画和子弹生成代码。
- 把“是否开火”的判断从玩家输入改为武器内部冷却和瞄准结果。

### 敌人死亡

当前敌人死亡与加分绑定。Gameplay 1 应改为：

- 敌人死亡
- 掉落经验
- 可选记录击杀数
- 不显示分数

## 第一轮最小交付范围

如果需要控制迭代规模，第一轮最小交付可以只包含：

1. 存活时间替代分数。
2. 初始武器自动射击。
3. 时间驱动持续刷怪。
4. 敌人死亡掉落经验。
5. 玩家拾取经验。
6. 经验满后显示 3 选 1。
7. 至少一种被动奖励生效。
8. 结算显示存活时间并返回主菜单。

这已经足够验证新的玩法方向。武器数量、被动数量和 UI 美术都可以后续扩展。

## 实现顺序总结

推荐严格按以下顺序推进：

1. 存活时间与结算替代分数。
2. 内容数据与配置访问入口。
3. 武器自动攻击。
4. 时间驱动刷怪。
5. 经验掉落与拾取。
6. 升级三选一。
7. 武器与被动持有列表。
8. 主菜单、关卡选择与结算流程。
9. 数值调优与最小打磨。

原因：

1. 先改局内目标，避免继续围绕分数扩展。
2. 先建立配置访问入口并划清数据边界，后续内容不需要反复改代码结构。
3. 自动攻击和刷怪先成立，经验与升级才有测试环境。
4. HUD 最后依据实际玩法状态改造，避免再次做错 UI。
